# Code Indexer – Technical Design (ContextMint)

## 1. Purpose and Scope

The **Code Indexer** is the pipeline that turns raw source repositories into structured, queryable artifacts for the System Knowledge Graph (SKG) and the Context Funnel. It does **not** parse source code directly; it consumes the output of the **Code Parser** and:

- Builds and maintains a **symbol graph** (services, modules, classes, functions, endpoints, dependencies).
- Produces **multi-level code representation** (Layer 0 raw, Layer 1 symbol graph, Layer 2 semantic summaries, Layer 3 architecture map).
- Writes normalized **nodes and edges** into the SKG.
- Feeds **chunks** and metadata to the **Vector Index** for semantic search.

The indexer is language-agnostic at the schema level: it works with a normalized model; language-specific parsing is delegated to the Code Parser (see `code-parser.md`).

---

## 2. High-Level Architecture

```text
                    Code Indexer (this component)
                                │
        ┌───────────────────────┼───────────────────────┐
        │                       │                       │
   Symbol Graph          Architecture Map       Semantic Summaries
   Builder               Builder                 (optional LLM)
        │                       │                       │
        └───────────────────────┼───────────────────────┘
                                │
                    Code Parser Output (AST / symbols)
                                │
        ┌───────────────────────┼───────────────────────┐
        │                       │                       │
   SKG Writer            Vector Index Writer      Cache / Incremental
   (graph nodes/edges)   (embeddings + metadata)  (delta index)
```

**Inputs**:
- Parsed repository output: list of **files**, **symbols** (modules, classes, functions, endpoints), **imports/dependencies**, **data access** (e.g. SQL, ORM usage), **message topics** (if detectable).
- Optional: project manifest (service name, repo path, language).

**Outputs**:
- Graph nodes and edges conforming to the SKG schema (Service, Endpoint, Function, Module, CALLS, IMPLEMENTS, READS, WRITES, etc.).
- Vector-indexable units: file-level or symbol-level chunks with metadata (path, symbol_id, service_id).
- Architecture map: high-level service → service and service → external dependency graph.
- Optional: semantic summaries per service/module/function (see Context Funnel and multi-level representation).

---

## 3. Multi-Level Code Representation

The indexer implements the **layered code representation** so the Context Funnel can load the right level of detail.

| Layer | Name                | Content                          | Typical size (tokens) | Use case                    |
|-------|---------------------|----------------------------------|------------------------|-----------------------------|
| 0     | Raw code            | Source files                     | Large (full file)      | Drill-down when needed      |
| 1     | Symbol graph        | Nodes + edges (no body text)     | Small (structural)     | Navigation, dependency flow |
| 2     | Semantic summaries  | AI-generated descriptions        | ~20–50 per symbol      | Quick understanding         |
| 3     | Architecture map   | Service topology + dependencies  | ~500–2k total          | System-wide context         |

**Indexer responsibilities by layer**:
- **Layer 0**: Store references (file path, range); optionally store content in blob store or leave on filesystem with path only.
- **Layer 1**: Build symbol graph from parser output; emit SKG nodes/edges; support incremental updates.
- **Layer 2**: Trigger or store semantic summaries (may be generated by a separate step/LLM); attach to graph nodes.
- **Layer 3**: Aggregate Layer 1 into service-level view; build architecture map (directed graph of services and external deps); persist as a dedicated structure or derived view in SKG.

## 4. Symbol Graph Schema (Output to SKG)

The indexer emits entities and relations that match the **normalized system model** in `Overview.md` and `knowledge-graph.md`.

**Node types (code-related)**:
- `Service` (logical deployable unit; may be 1:1 with repo or a slice of a monorepo).
- `Module` (package / namespace / directory representing a unit of code).
- `Class` (optional; language-dependent).
- `Function` (procedure, method, handler).
- `Endpoint` (HTTP route, RPC method, queue consumer — whatever the parser can detect).

**Edge types**:
- `Service -[:CONTAINS]-> Module`
- `Module -[:CONTAINS]-> Class | Function`
- `Class -[:CONTAINS]-> Function`
- `Service -[:IMPLEMENTS]-> Endpoint`
- `Function -[:IMPLEMENTS]-> Endpoint` (e.g. handler function for POST /login)
- `Function -[:CALLS]-> Function`
- `Module -[:IMPORTS]-> Module` (or cross-service dependency)
- `Function -[:READS]-> DatabaseTable` (from parser/DB ingestion)
- `Function -[:WRITES]-> DatabaseTable`
- `Service -[:CALLS]-> Service` (derived from HTTP client calls, RPC, or config)

**Attributes** (examples):
- `Service`: `name`, `repo_path`, `language`, `project_id`
- `Function`: `name`, `file_path`, `line_start`, `line_end`, `signature`, `summary` (if Layer 2)
- `Endpoint`: `method`, `path`, `service_id`

## 5. Indexing Pipeline Stages

1. **Ingest**
   - Receive parser output (per file or per repo batch).
   - Validate and normalize (e.g. resolve paths, deduplicate symbols).

2. **Service / Module Boundary Resolution**
   - Map files and modules to a **Service** (from manifest or heuristics: e.g. one service per repo, or per directory in a monorepo).
   - Create or update `Service` and `Module` nodes.

3. **Symbol Graph Build**
   - Create `Class`, `Function`, `Endpoint` nodes.
   - Create edges: CONTAINS, IMPLEMENTS, CALLS, IMPORTS.
   - Resolve cross-file references (e.g. CALLS to a function in another file) using symbol IDs or qualified names.

4. **Data Access and External Call Extraction**
   - Use parser output (e.g. SQL strings, ORM calls, HTTP client calls) to create READS/WRITES and CALLS edges.
   - May be augmented by **data-ingestion** (DB schema) to attach to `DatabaseTable` nodes.

5. **Architecture Map**
   - Aggregate to service-level: which services call which (and external APIs/caches).
   - Persist as a compact structure (e.g. adjacency list or dedicated graph view).

6. **Vector Chunking**
   - Slice code into chunks (e.g. by function or by fixed line window with overlap).
   - Attach metadata: `symbol_id`, `service_id`, `file_path`, `layer`.
   - Send to Vector Index writer (see `vector-index.md`).

7. **SKG Write**
   - Write all nodes and edges to the graph store (batch or transactional).
   - Support **incremental** mode: only changed files/symbols (requires diff from parser or content hashing).

8. **Optional: Semantic Summaries (Layer 2)**
   - For each Service, Module, or Function, either:
     - Read from cache (if already generated), or
     - Call summarization service (LLM) and store result on the node or in a separate store keyed by symbol_id.

---

## 6. Incremental and Full Re-Index

- **Full index**: Process entire repo; rebuild symbol graph and architecture map; refresh vector chunks.
- **Incremental**:
  - Input: list of changed files (e.g. from git diff or file watcher).
  - Re-run parser only for those files (or affected modules).
  - Compute diff: new/updated/deleted symbols.
  - Update SKG: upsert nodes/edges, delete obsolete edges/nodes.
  - Update vector index: add/update/delete chunks for affected regions.
  - Recompute architecture map if any service boundary or cross-service call changed.

**Triggers**:
- Webhook from Git (e.g. on push).
- Scheduled full re-index (nightly).
- CLI: `ctmint index [--incremental]`.

---

## 7. Integration Points

- **Code Parser** (`code-parser.md`): Provides AST/symbol list per file; indexer consumes this and never reads raw source for structure (only optionally for Layer 0 storage or chunking).
- **Knowledge Graph** (`knowledge-graph.md`): Indexer is the main writer for code-related nodes and edges; must conform to SKG schema and ID conventions.
- **Vector Index** (`vector-index.md`): Indexer produces chunks + metadata; vector index stores embeddings and supports scope-limited search (e.g. by `service_id`).
- **Orchestrator / Context Funnel** (`orchestrator.md`, `context-funnel.md`): Read-only consumers; query SKG and vector index for architecture map, symbol graph, summaries, and code snippets.

---

## 8. Configuration and Deployment

- **Config** (per project or global):
  - Repo path(s), service boundaries (e.g. one service per repo, or mapping of dir → service name).
  - Chunking strategy: by symbol, by line window, overlap size.
  - Whether to run Layer 2 summarization (and endpoint for LLM).
  - Retention: how long to keep Layer 0 blobs, whether to keep raw content at all.
- **Deployment**: Runs inside the same binary (Rust) as part of the single-binary core; can be invoked via CLI (`ctmint index`) or by an internal scheduler/webhook handler.

---

## 9. Error Handling and Observability

- **Partial failure**: If a single file fails to parse, log and skip; continue with rest of repo; report failed files in index report.
- **Consistency**: Prefer transactional or batch writes to SKG so that a run is either fully applied or not (or use versioning so that readers see a consistent snapshot).
- **Metrics**: Index duration, number of files/symbols processed, number of SKG nodes/edges written, vector chunks written; optional: latency per stage.
- **Logs**: Structured logs for each stage (ingest, graph build, SKG write, vector write) to aid debugging and auditing.

---

## 10. Summary

The Code Indexer turns **Code Parser** output into **SKG-compliant symbol graph**, **architecture map**, and **vector index input**, and optionally **semantic summaries**. It implements the multi-level code representation (L0–L3) so that the rest of ContextMint can navigate and retrieve code at the right granularity without loading entire repositories into the LLM.
